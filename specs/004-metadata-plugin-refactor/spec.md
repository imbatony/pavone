## 用户场景与测试 *(必填)*

### 用户故事 1 - 新插件开发效率提升 (优先级: P1)

作为一名插件开发者，我希望在开发新的元数据提取插件时，只需关注核心解析逻辑（识别、解析），而不必重复编写 URL 解析、页面获取、错误处理、日期/时长格式化等样板代码，从而将新插件开发成本降低约 44%。

**优先级原因**: 这是本次重构的核心价值——通过消除结构性重复，直接降低每个新插件的开发和维护成本。

**独立测试**: 可以通过编写一个新的示例插件来验证：继承新基类后，仅实现 `can_extract`、`_resolve` 和 `_parse` 三个方法即可完成完整的元数据提取功能。

**验收场景**:

1. **给定** 新的 `HtmlMetadataPlugin` 基类已就绪, **当** 开发者创建新的 HTML 解析型插件时, **那么** 只需实现 `can_extract`、`_resolve`、`_parse` 三个方法即可完成基本功能
2. **给定** 新的 `ApiMetadataPlugin` 基类已就绪, **当** 开发者创建新的 API 型插件时, **那么** 只需实现 `can_extract`、`_resolve`、`_parse` 三个方法即可完成基本功能
3. **给定** 基类提供公共工具方法, **当** 开发者需要解析日期、时长或处理相对 URL 时, **那么** 可直接调用基类的 `_parse_date`、`_parse_runtime`、`_abs` 等方法，无需重复编写

### 边界情况

- 当插件的 `_parse_runtime` 实现与基类通用版本有细微差异时（如特定站点使用非标准时长格式），该插件需覆写该方法以保持行为一致。迁移时逐一对比确认。
- 当插件的 `_resolve` 方法返回的 URL 需要特殊处理（如添加查询参数、设置 Referer）时，可通过覆写 `_fetch_page` 钩子方法或在 `_resolve` 中直接处理。
- 当 `MetadataPlugin` 基类未来新增接口时，中间层基类（`HtmlMetadataPlugin` 等）须保持向上兼容，不阻断继承链。
- 当某个插件同时使用 HTML 解析和 API 调用（如 heyzo 的 JSON-LD + HTML fallback）时，应根据主要解析方式选择基类，次要方式通过覆写实现。
- 当基类的公共方法被修改时，须通过全量测试覆盖保障所有已迁移插件的行为不受意外影响。

## 需求 *(必填)*

### 功能需求

**基类提取与架构设计**

- **FR-001**: 系统必须在 `MetadataPlugin` 和具体 HTML 解析插件之间引入 `HtmlMetadataPlugin` 中间基类，封装 HTML 类插件的通用流程（resolve → fetch → parse）和统一错误处理
- **FR-002**: 系统必须引入 `ApiMetadataPlugin` 中间基类，封装 API/JSON 类插件的通用流程和错误处理
- **FR-003**: 系统必须引入 `JsonLdMetadataPlugin` 中间基类，封装 JSON-LD 解析类插件的通用流程
- **FR-004**: `HtmlMetadataPlugin` 必须提供 `extract_metadata` 模板方法，实现 resolve → fetch → parse 的标准流程，子类无需重复编写此模板
- **FR-005**: `HtmlMetadataPlugin` 必须提供 `_fetch_page` 钩子方法，允许子类覆写以自定义页面获取行为（如添加 cookies、headers、特殊超时设置）

**公共工具方法统一**

- **FR-006**: 基类必须提供统一的 `_abs` 方法（相对 URL 转绝对 URL），替代当前分散在各插件中的 17 份重复实现
- **FR-007**: 基类必须提供统一的 `_parse_runtime` 方法（时长文本转分钟数），替代当前分散在各插件中的 14 份重复实现
- **FR-008**: 基类必须提供统一的 `_parse_date` 方法（日期文本转标准格式），替代当前分散在各插件中的 23 份重复实现
- **FR-009**: 基类必须提供统一的 `_parse_iso_duration` 方法（ISO 8601 时长转分钟数），供 JSON-LD 类插件使用
- **FR-010**: 基类必须新增 `_get_tag_attr` 类型安全工具方法，安全获取 BeautifulSoup 标签属性并返回 `Optional[str]` 类型

**插件迁移**

- **FR-011**: 所有适用的 HTML 解析型插件必须迁移至 `HtmlMetadataPlugin` 基类，删除本地重复的 `extract_metadata` 模板代码和公共工具方法
- **FR-012**: 所有 API/JSON 型插件（modelmediaasia, muramura 等）必须迁移至 `ApiMetadataPlugin` 基类
- **FR-013**: 所有 JSON-LD 型插件（c0930, h0930, h4610 等）必须迁移至 `JsonLdMetadataPlugin` 基类
- **FR-014**: 迁移必须分批进行（至少 6 批），每批独立提交，每批迁移后全量测试通过

**行为兼容性**

- **FR-015**: 迁移后每个插件的 `extract_metadata` 输出必须与迁移前完全一致（给定相同输入，返回相同的元数据字段值）
- **FR-016**: 迁移后每个插件的错误处理行为必须与迁移前一致（无效输入返回 None，不抛出未处理异常）
- **FR-017**: `MetadataPlugin` 基类的公开接口必须保持不变，未迁移的插件和第三方插件不受影响

**灵活性保留**

- **FR-018**: 具有特殊流程的插件（如 avbase 的 API+HTML 双通道、fanza 的 `__NEXT_DATA__` 解析）可选择直接继承 `MetadataPlugin` 或覆写基类方法，不被强制适配标准模板
- **FR-019**: 子类必须能够覆写基类的任何钩子方法（`_fetch_page`、`extract_metadata` 等）以实现差异化行为

### 关键实体

- **MetadataPlugin**: 已有的抽象基类，定义 `can_extract` 和 `extract_metadata` 两个抽象方法。重构后接口保持不变，作为整个插件继承树的根节点。
- **HtmlMetadataPlugin**: 新增中间基类，继承自 `MetadataPlugin`。封装 HTML 页面类插件的通用流程（resolve → fetch → parse）、错误处理模板和公共工具方法。子类只需实现 `can_extract`、`_resolve`、`_parse`。
- **ApiMetadataPlugin**: 新增中间基类，继承自 `MetadataPlugin`。封装 API/JSON 类插件的通用流程。子类只需实现 `can_extract`、`_resolve`、`_parse`。
- **JsonLdMetadataPlugin**: 新增中间基类，继承自 `HtmlMetadataPlugin`。在 HTML 解析基础上增加 JSON-LD 数据自动提取能力。
- **FC2BaseMetadata**: 已有中间基类，重构后改为继承 `HtmlMetadataPlugin`（原继承 `MetadataPlugin`）。为 fc2hub 和 fc2ppvdb 提供 FC2 专属公共逻辑，同时享受新基类的公共方法和模板。
- **具体插件**: 34 个已有的元数据提取插件，根据解析方式分别继承上述中间基类之一，或在特殊情况下直接继承 `MetadataPlugin`。

## 假设

- 当前 34 个插件中的 `_abs`、`_parse_runtime`、`_parse_date` 等方法实现虽然分散但逻辑基本一致，可以安全统一到基类中。采用**超集策略**：基类方法合并处理所有已知格式变体（各变体互不冲突，不产生解析歧义），插件无需覆写。
- 大多数 HTML 解析型插件遵循 resolve → fetch → parse 的标准流程，适合使用模板方法模式统一。
- BeautifulSoup 的 HTML 解析器统一使用 `lxml`，这是当前所有插件已采用的解析器。
- `JsonLdMetadataPlugin` 继承自 `HtmlMetadataPlugin`（而非并列关系），因为 JSON-LD 插件都需要先获取 HTML 页面再提取 JSON-LD 数据，本质是 HTML 解析的扩展。
- 已有的 `FC2BaseMetadata` 中间基类改为继承 `HtmlMetadataPlugin`，继承链变为：MetadataPlugin → HtmlMetadataPlugin → FC2BaseMetadata → Fc2HubMetadata / Fc2PpvdbMetadata，统一继承链同时保留 FC2 专属公共逻辑。
- 分 6 批迁移的顺序基于插件复杂度从低到高排列，降低每批的风险。
- 本版本交付范围为阶段一（基类提取）和阶段二（全量插件迁移）。阶段三的可选优化（通用 HTML 字段解析器、MetadataBuilder 简化、C901 复杂度降低）不属于本规范交付范围，作为后续版本候选项。
- 行为兼容性通过现有 Mock 测试验证：现有测试已使用固定 HTML fixture 响应，迁移后运行相同测试即可确认行为一致性，无需额外快照机制。
