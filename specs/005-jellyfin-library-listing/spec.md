# 功能规范: Jellyfin 媒体库视频列表

**功能分支**: `005-jellyfin-library-listing`
**创建时间**: 2026-04-17
**状态**: 草稿
**输入**: 用户描述: "添加Jellyfin媒体库视频列表命令行功能，支持多种排序方式、元数据丰富度评分、交互式媒体库选择和表格显示"

## 用户场景与测试 *(必填)*

### 用户故事 1 - 指定媒体库列取视频并排序 (优先级: P1)

作为一名使用 Jellyfin 管理视频的用户，我希望能够通过命令行列取指定媒体库中的所有视频，并支持按名称、加入时间等字段排序，以便快速浏览和查找媒体库中的内容。

**优先级原因**: 列取视频是本功能的核心能力，所有其他功能（评分排序、交互选择）都建立在此基础之上。没有基础列取能力，其他功能无法运作。

**独立测试**: 可以通过指定一个已存在的媒体库名称运行列取命令，验证输出的表格内容和排序结果，完全独立交付浏览媒体库内容的价值。

**验收场景**:

1. **给定** 用户已配置好 Jellyfin 连接且存在至少一个含视频的媒体库, **当** 用户执行列取命令并指定媒体库名称时, **那么** 系统以表格形式展示该媒体库中的视频列表，包含序号、名称、加入时间、元数据丰富度评分、视频地址五列
2. **给定** 用户指定了排序字段为"名称", **当** 执行列取命令时, **那么** 视频按名称的字母/拼音顺序排列
3. **给定** 用户指定了排序字段为"加入时间", **当** 执行列取命令时, **那么** 视频按加入系统的时间排列（默认最新在前）
4. **给定** 用户指定了降序排列, **当** 执行列取命令时, **那么** 列表按指定字段的降序展示
5. **给定** 媒体库中视频数量超过单页显示量, **当** 执行列取命令时, **那么** 默认显示前 50 条记录，用户可通过参数调整显示数量

---

### 用户故事 2 - 元数据丰富度评分与排序 (优先级: P2)

作为一名重视视频元数据完整性的用户，我希望系统能够根据每个视频的元数据丰富程度计算评分，并支持按此评分排序，以便快速识别哪些视频的元数据需要补充完善。

**优先级原因**: 元数据丰富度评分是区别于 Jellyfin 内置排序能力的差异化功能，帮助用户识别元数据缺失的视频，直接支持内容管理工作流。

**独立测试**: 可以通过列取视频并指定按元数据评分排序，验证评分计算逻辑和排序结果是否正确，独立交付元数据质量审查的价值。

**验收场景**:

1. **给定** 媒体库中存在元数据完整度不同的视频, **当** 用户指定按元数据评分排序时, **那么** 系统按评分高低排列视频
2. **给定** 一个视频拥有完整的标题、简介、类型、演员、封面图、年份等元数据, **当** 系统计算其评分时, **那么** 该视频获得较高的丰富度评分
3. **给定** 一个视频仅有标题和文件路径而缺少其他元数据, **当** 系统计算其评分时, **那么** 该视频获得较低的丰富度评分
4. **给定** 用户指定按评分升序排列, **当** 执行列取命令时, **那么** 元数据最不完整的视频排在最前面，方便用户优先补充

---

### 用户故事 3 - 交互式选择媒体库 (优先级: P2)

作为一名拥有多个媒体库的用户，当我执行列取命令时未指定具体的媒体库，我希望系统能提供交互式选择界面让我从已有媒体库中选择，而不是直接报错退出。

**优先级原因**: 交互式选择显著降低使用门槛，用户无需记忆媒体库名称即可操作，提升使用体验。与基础列取功能同等重要。

**独立测试**: 可以通过不指定媒体库名称运行列取命令，验证交互式选择列表是否正确展示所有可用媒体库，选择后是否正确列取对应内容。

**验收场景**:

1. **给定** Jellyfin 上存在多个媒体库且用户未指定媒体库名称, **当** 用户执行列取命令时, **那么** 系统展示所有可用媒体库的编号列表供用户选择
2. **给定** 交互式选择界面已展示, **当** 用户输入对应编号时, **那么** 系统列取该媒体库的视频
3. **给定** 用户输入了无效的编号, **当** 系统接收到输入时, **那么** 提示输入无效并要求重新选择
4. **给定** Jellyfin 上仅有一个媒体库且用户未指定名称, **当** 用户执行列取命令时, **那么** 系统自动选择该唯一媒体库而不弹出选择界面

---

### 用户故事 4 - 表格化展示结果 (优先级: P1)

作为用户，我希望视频列表以整齐的表格形式展示，包含名称、加入时间、元数据丰富度评分、视频地址等关键列，以便快速扫视和对比信息。

**优先级原因**: 表格是信息展示的基础形式，直接影响用户阅读体验和信息获取效率，是核心展示层能力。

**独立测试**: 可以通过任意有效的列取命令验证输出表格的格式、对齐、列宽、中文字符处理等是否正确。

**验收场景**:

1. **给定** 列取结果包含多条视频记录, **当** 系统展示表格时, **那么** 表格包含表头行、分隔线和数据行，列包含序号、名称、加入时间、元数据评分、视频路径
2. **给定** 视频名称包含中文字符, **当** 系统展示表格时, **那么** 中文字符的列宽计算正确，表格对齐不错位
3. **给定** 视频名称或路径过长, **当** 系统展示表格时, **那么** 过长内容被截断并以省略号标识，不破坏表格整体布局
4. **给定** 列表为空（媒体库无视频）, **当** 系统展示时, **那么** 显示友好的提示信息而非空表格

---

### 边界情况

- 当 Jellyfin 服务器不可达或认证失败时，系统应给出明确的连接错误提示
- 当用户指定的媒体库名称不存在时，提示库不存在并列出可用的媒体库名称
- 当媒体库为空（无任何视频）时，显示空库提示而不是错误
- 当网络延迟导致获取视频列表超时时，给出超时提示
- 当视频缺少路径信息（如远程/流媒体内容）时，路径列显示为"无"
- 当排序字段指定了不支持的值时，提示有效的排序选项

## 需求 *(必填)*

### 功能需求

- **FR-001**: 系统必须提供列取指定 Jellyfin 媒体库中所有视频的命令，以表格形式展示结果
- **FR-002**: 系统必须支持按以下字段排序视频列表：名称（name）、加入时间（date_added）、元数据丰富度评分（metadata_score）
- **FR-003**: 系统必须支持升序和降序两种排列方向，默认降序（最新/最高在前）
- **FR-004**: 系统必须为每个视频计算元数据丰富度评分，评分基于以下元数据字段的填充情况：标题、简介（overview）、类型（genres）、年份（year）、演员（actors）、导演（directors）、封面图（primary image）、评分（rating）、标签（tags）、工作室（studios）
- **FR-005**: 元数据丰富度评分以百分制呈现（0-100），每个元数据维度占一定权重，有内容得分，无内容不得分
- **FR-006**: 当用户未通过命令行参数指定媒体库时，系统必须展示所有可用媒体库的编号列表，允许用户交互式选择
- **FR-007**: 当仅存在一个媒体库且用户未指定时，系统自动选择该唯一媒体库
- **FR-008**: 表格必须包含以下列：序号、名称、加入时间、元数据评分、视频路径
- **FR-009**: 表格必须正确处理中文字符的列宽对齐
- **FR-010**: 过长的名称或路径应被截断并附加省略号，保持表格布局整齐
- **FR-011**: 系统必须支持限制显示记录数的参数，默认显示 50 条
- **FR-012**: 当指定的媒体库名称不存在时，系统应提示错误并列出可用的媒体库名称
- **FR-013**: 当媒体库为空时，显示友好的空结果提示

### 关键实体

- **媒体库（Library）**: Jellyfin 中的内容组织单位，包含名称、唯一标识、类型和项目数量
- **视频项（Video Item）**: 媒体库中的单个视频条目，包含名称、文件路径、加入时间和各类元数据
- **元数据丰富度评分（Metadata Score）**: 基于视频元数据各维度填充情况的计算值，反映元数据完整程度

## 成功标准 *(必填)*

### 可衡量的结果

- **SC-001**: 用户可以在一条命令内列取指定媒体库中的视频列表并查看关键信息
- **SC-002**: 用户可以通过不同排序方式（名称、时间、评分）快速定位目标视频
- **SC-003**: 用户在未指定媒体库时，可以在 3 步交互内（查看列表→输入编号→获得结果）完成操作
- **SC-004**: 元数据丰富度评分能准确区分元数据完整和缺失的视频，评分差异至少 30 分
- **SC-005**: 表格在包含中文字符和长路径时仍保持列对齐，不出现错位
- **SC-006**: 包含 1000 个视频的媒体库能在合理时间内完成列取和展示

## 假设

- Jellyfin 服务器连接和认证配置已正确设置（复用现有配置体系）
- 元数据丰富度评分的维度和权重采用内置默认值，无需用户自行配置
- 视频路径指的是 Jellyfin 服务器上的文件系统路径
- 排序操作在客户端完成（对于元数据评分排序），Jellyfin 原生排序字段可利用服务端排序
- 默认每页 50 条是合理的显示量，满足大部分使用场景
