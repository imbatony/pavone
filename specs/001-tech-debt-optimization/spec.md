## 用户场景与测试 *(必填)*

### 用户故事 1 - Ctrl+C 优雅终止下载 (优先级: P1)

作为一个正在下载视频的用户, 我希望在按下 Ctrl+C 后程序能够立即响应并优雅退出, 而不是卡死或需要多次按 Ctrl+C 甚至强制杀进程. 退出时程序应清理临时资源, 并保留已下载的分片以便下次断点续传.

**优先级原因**: 这是一个阻塞性的可用性 Bug — 用户无法正常终止运行中的下载任务, 严重影响日常使用体验. 当前按 Ctrl+C 后程序无法及时响应, 用户被迫强制杀进程, 可能导致临时文件泄漏和磁盘空间浪费.

**独立测试**: 可以通过启动一个 M3U8 或 HTTP 下载任务, 在下载过程中按 Ctrl+C, 验证程序在 3 秒内退出且不产生未清理的临时文件.

**验收场景**:

1. **给定** 用户正在下载一个 M3U8 视频 (含 50+ 分片), **当** 用户按下 Ctrl+C, **那么** 程序在 3 秒内停止所有下载线程, 输出"下载已中断"提示, 并正常退出 (exit code 130)
2. **给定** 用户正在执行 HTTP 并发分片下载, **当** 用户按下 Ctrl+C, **那么** 所有活动线程停止, 已下载的分片文件保留在缓存目录中, 程序正常退出
3. **给定** 用户中断了一个 M3U8 下载后重新执行相同下载命令, **当** 程序检测到缓存中已有部分分片, **那么** 自动跳过已下载的分片, 从断点继续下载

---

### 用户故事 2 - M3U8 分段失败可跳过 (优先级: P1)

作为一个下载长视频的用户, 我希望当部分视频分片下载失败时, 程序询问我是否跳过失败分片继续合并, 而不是直接宣告整个下载失败. 这样即使有少量分片丢失, 我仍然可以获得一个基本完整的视频文件.

**优先级原因**: 当前任何一个分片失败就导致整个下载失败, 对于几百个分片的长视频这是不可接受的. 网络波动导致 1-2 个分片失败是常见情况, 用户应当有权选择跳过这些分片而不是从头重来.

**独立测试**: 可以通过模拟一个包含故意失败分片的 M3U8 下载任务, 验证系统提示用户选择, 以及选择"跳过"后成功合并剩余分片.

**验收场景**:

1. **给定** 用户正在下载一个 50 段的 M3U8 视频, 其中 2 段在重试后仍然失败, **当** 所有分片尝试完成后, **那么** 系统显示失败分片的编号和总数 (如 "2/50 段下载失败"), 并提示用户选择: 重试失败分片 / 跳过失败分片并合并 / 取消下载
2. **给定** 用户选择"跳过失败分片并合并", **当** 系统执行合并操作, **那么** 系统使用已成功下载的 48 个分片生成视频文件, 并在完成时报告实际合并的分片数量
3. **给定** 用户选择"重试失败分片", **当** 系统重新下载失败的 2 个分片, **那么** 仅重新下载失败分片 (不重复下载已成功的分片), 重试后再次汇报结果

### 用户故事 3 - M3U8 进度条优化 (优先级: P2)

作为一个下载 M3U8 视频的用户, 我希望看到清晰的分片级别进度信息, 包括已完成分片数/总分片数、预计剩余时间和下载速度, 而不是仅仅看到一个不断增长的字节数和无法计算百分比的进度条.

**优先级原因**: 当前 M3U8 下载时进度条无法显示百分比和预估完成时间, 用户对下载进展毫无概念. 虽然不影响功能正确性, 但严重影响使用信心和体验.

**独立测试**: 可以通过下载一个多分片 M3U8 视频, 观察进度条是否显示"已完成/总数"格式的分片进度、百分比、速度和预估剩余时间.

**验收场景**:

1. **给定** 用户开始下载一个 100 段的 M3U8 视频, **当** 下载进行到第 30 段, **那么** 进度条显示类似 "30/100 段 | 30% | 2.5 段/秒 | 剩余约 28 秒" 的信息
2. **给定** 用户正在下载 M3U8 视频且部分分片从缓存恢复, **当** 进度条更新, **那么** 恢复的分片计入已完成数量, 进度条从恢复点开始而非从 0 开始
3. **给定** 用户同时能看到 M3U8 和 HTTP 两种下载的进度, **当** 对比两种进度显示, **那么** 两者风格一致 (均使用相同的进度条框架), 但 M3U8 显示分片计数, HTTP 显示字节量

---

### 用户故事 4 - 插件自动发现与加载 (优先级: P2)

作为一个开发者, 我希望新增一个插件时只需要创建插件类文件并继承正确的基类, 系统就能自动发现和加载它, 而不需要手动在多处代码中注册导入. 这减少了添加插件时遗忘注册步骤的风险, 也降低了新贡献者的学习成本.

**优先级原因**: 当前添加一个新插件需要手动修改多处注册代码和导入语句, 容易遗漏且违反开放-封闭原则. 作为插件化架构的核心, 自动发现机制是降低维护成本和新贡献者入门门槛的关键改进.

**独立测试**: 可以通过在插件目录下新增一个测试插件文件 (继承 ExtractorPlugin), 不修改任何注册代码, 验证系统启动后能自动发现并加载该插件.

**验收场景**:

1. **给定** 开发者在插件目录下创建了一个新的插件类 (继承 `ExtractorPlugin`), **当** 系统启动并初始化插件管理器, **那么** 新插件被自动发现、实例化并注册, 无需修改任何其他文件
2. **给定** 系统配置中某个插件名被加入禁用列表, **当** 系统启动, **那么** 该插件被发现但不被加载, 且在 verbose 模式下记录跳过原因
3. **给定** 插件目录中存在一个有语法错误的插件文件, **当** 系统启动, **那么** 该文件的加载失败被记录为警告, 其他正常插件不受影响

---

### 用户故事 5 - 消除重复代码, 抽取公共工具类 (优先级: P3)

作为一个维护者, 我希望各插件中重复的 HTML 元数据提取逻辑 (标题、番号、封面、演员、日期、M3U8 链接提取) 被提炼到共享的工具类中, 这样修复一个提取 Bug 只需改一处, 新增插件时也能复用已有的提取能力.

**优先级原因**: 当前 5 个插件各自实现了高度相似的 `_extract_*` 方法, 约 500+ 行重复代码. 虽然不影响当前功能, 但显著增加了维护成本和 Bug 修复的遗漏风险. 此项优先级较低是因为它是纯内部重构, 不直接改善用户体验.

**独立测试**: 可以通过运行现有的全部插件测试, 验证重构后所有插件的提取结果与重构前完全一致.

**验收场景**:

1. **给定** 系统中存在统一的元数据提取工具, **当** 新增一个插件需要从 HTML 中提取标题和番号, **那么** 开发者可以调用共享提取方法并传入站点特定的选择器/正则, 无需从其他插件复制代码
2. **给定** 提取工具中修复了一个日期解析 Bug, **当** 所有使用该日期解析的插件处理相同的输入, **那么** 所有插件都获得修复, 无需逐个修改
3. **给定** 所有现有插件测试通过, **当** 共享提取逻辑替换了各插件的内联实现后运行完整测试套件, **那么** 所有测试仍然通过, 无回归

---

### 边界情况

- 当所有 M3U8 分片都失败时, 系统应直接报告下载失败而不是提示"跳过"(没有任何分片可合并)
- 当用户在交互式提示 (选择跳过/重试/取消) 等待期间按 Ctrl+C, 系统应当立即退出
- 当插件自动发现过程中发现类名冲突 (两个插件使用相同名称), 系统应记录错误并加载优先级更高的那个
- 当 M3U8 下载被 Ctrl+C 中断后重新启动, 进度条应正确反映从缓存恢复的分片数量
- 当重构后的共享工具类与某个特定站点的 HTML 结构不兼容, 插件应能 fallback 到自定义提取逻辑

## 需求 *(必填)*

### 功能需求

**Ctrl+C 优雅终止**:

- **FR-001**: 系统必须注册操作系统信号处理器, 在收到中断信号后设置全局中断标志
- **FR-002**: 所有活动的并发下载线程必须在中断标志被设置后停止接受新任务, 并让当前任务尽快完成或终止
- **FR-003**: 系统在中断后必须保留已完整下载的缓存分片和正在传输中的 .part 文件. 下次启动时必须通过文件大小 + Range 请求验证 .part 文件是否可续传, 若不可续传则删除重下
- **FR-004**: 系统在中断退出前必须向用户显示中断确认消息

**M3U8 分段失败处理**:

- **FR-005**: 系统必须在所有分片下载尝试完成后, 汇总成功和失败的分片信息
- **FR-006**: 当存在失败分片时, 系统必须向用户展示失败详情并提供交互式选择: 重试 / 跳过并合并 / 取消. 在非交互式环境 (stdin 不可用) 中, 默认行为为自动取消下载
- **FR-006a**: 系统必须提供 `--skip-failed` CLI 标志, 允许用户预先声明跳过策略, 在非交互式环境中无需等待用户输入即可跳过失败分片并继续合并
- **FR-007**: 系统必须支持仅重试失败分片 (不重复下载已成功的分片)
- **FR-008**: 系统必须支持跳过失败分片, 使用已成功的分片执行合并操作
- **FR-009**: 当所有分片均失败时, 系统必须直接报告失败, 不提供"跳过并合并"选项

**M3U8 进度条优化**:

- **FR-010**: M3U8 下载过程中的进度显示必须包含: 已完成分片数、总分片数、完成百分比
- **FR-011**: 进度显示必须包含下载速度 (主显示分片/秒, verbose 模式下辅助显示字节/秒) 和基于分片速度计算的预估剩余时间
- **FR-012**: 从缓存恢复的分片必须计入已完成数量, 进度条从实际起点开始显示

**插件自动发现**:

- **FR-013**: 系统必须能在指定的插件包下自动扫描并发现所有继承自 BasePlugin 的类
- **FR-014**: 自动发现的插件必须根据配置中的禁用列表进行过滤
- **FR-015**: 插件发现过程中的单个插件加载失败必须被隔离, 不得影响其他插件的加载
- **FR-016**: 自动发现机制启用后, 必须完全移除硬编码的插件注册映射表和对应的手动导入语句, 不保留 fallback 机制

**代码质量优化**:

- **FR-017**: 各插件中重复的 HTML 元数据提取方法必须提炼为按提取类型分离的独立工具函数 (如 `extract_title`, `extract_actors`, `extract_cover`, `extract_date` 等), 插件按需组合调用
- **FR-018**: 每个独立提取函数必须支持通过参数化 (选择器、正则表达式) 适配不同站点的 HTML 结构
- **FR-019**: 各插件必须保留覆盖共享提取逻辑的能力, 以应对非标准的站点结构

### 关键实体

- **中断标志 (Interruption Flag)**: 全局可见的布尔状态, 标识用户是否请求了中断. 所有下载线程和执行管理器检查此标志决定是否继续工作.
- **分片下载结果 (Segment Result)**: 记录每个 M3U8 分片的下载结果, 包含: 分片索引、下载状态 (成功/失败)、失败原因. 用于汇总报告和选择性重试.
- **分片进度信息 (Segment Progress)**: 扩展的进度信息, 除字节级数据外, 还包含总分片数和已完成分片数, 用于 M3U8 特有的进度显示.

## Clarifications

### Session 2026-03-16

- Q: 非交互式环境 (管道、CI、定时任务) 中 M3U8 分片失败时的默认行为是什么? → A: 默认自动取消; 提供 `--skip-failed` CLI 标志预声明跳过策略
- Q: 插件自动发现的迁移策略 — 是纯自动发现还是保留硬编码作为 fallback? → A: 完全移除硬编码注册, 纯自动发现; 若自动发现失败则核心插件也不可用
- Q: 共享提取工具的抽象粒度 — 统一提取器类还是独立工具函数? → A: 按提取类型分离为独立工具函数 (`extract_title`, `extract_actors` 等), 插件按需组合调用
- Q: HTTP 多线程下载中断时正在传输的 .part 文件如何处理? → A: 保留 .part 文件; 下次启动时通过文件大小 + Range 请求验证是否可续传, 不可续传则删除重下
- Q: M3U8 进度条的速度度量单位是分片/秒还是字节/秒? → A: 主显示分片/秒 (与百分比和 ETA 直接关联); 字节/秒作为 verbose 模式下的辅助信息

## 假设

- 章程中已定义的 `BasePlugin` 抽象契约和 `cleanup()` 资源清理方法将作为本次优化的基础, 不进行重新设计
- 当前 `HTMLMetadataExtractor`、`MetadataBuilder`、`OperationItemBuilder` 等已有工具类将作为重构的起点, 而非全部推翻重写
- M3U8 分片合并工具 (ffmpeg 或内置合并) 能够处理非连续分片的拼接 (跳过缺失的分片索引)

- **SC-005**: 各插件中的重复元数据提取代码减少 60% 以上, 同时所有现有测试保持通过
- **SC-006**: 重构后的代码通过 Pyright standard 模式检查, 无新增类型错误
