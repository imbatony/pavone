# 功能规范: MetaTube 元数据提取器移植

**功能分支**: `003-metatube-metadata-extractor`
**创建时间**: 2026-04-13
**状态**: 草稿
**输入**: 用户描述: "开启新的版本0.4.0，移植MetaTube里面的元数据提取器，我已经把源码下载到D:\\code\\metatube-sdk-go-main，其中针对各个网站的元数据提取功能在provider文件夹下，注意不要改动已经有的元数据提取器"

## 用户场景与测试 *(必填)*

### 用户故事 1 - 支持更多站点元数据自动提取 (优先级: P1)

作为开发者/用户，我希望系统能集成MetaTube中provider目录下的各站点元数据提取能力（不覆盖已有同类提取器），以便在处理多站点视频时自动获取丰富的元数据信息。

**优先级原因**: 直接提升系统对多站点的兼容性和元数据丰富度，减少手动维护和重复开发。

**独立测试**: 针对每个新移植的站点，输入对应站点的视频链接，系统能正确返回结构化元数据。

**验收场景**:
1. **给定** 已集成MetaTube provider下某站点提取器，**当** 输入该站点视频链接，**那么** 返回结构化元数据（如标题、演员、标签等）。
2. **给定** 已有的元数据提取器，**当** 输入已支持站点链接，**那么** 仍然优先使用原有实现，结果不变。

---

### 用户故事 2 - 便于后续扩展和维护 (优先级: P2)

作为开发者，我希望移植后的提取器结构清晰、易于维护，便于后续添加新站点或修复问题。

**优先级原因**: 降低维护成本，提升代码可扩展性。

**独立测试**: 新增或调整某站点提取逻辑时，不影响其他站点功能，且能快速定位和修复问题。

**验收场景**:
1. **给定** 新增/修改某站点提取器，**当** 运行测试，**那么** 其他站点提取功能不受影响。

---

### 用户故事 3 - 兼容性与错误处理 (优先级: P3)

作为用户，我希望系统在遇到不支持或异常站点时能给出明确提示，不影响整体流程。

**优先级原因**: 提升用户体验，避免因单一站点异常影响整体。

**独立测试**: 输入不支持或异常站点链接，系统能返回友好错误信息。

**验收场景**:
1. **给定** 输入未集成的站点链接，**当** 触发元数据提取，**那么** 返回“暂不支持”或类似提示。

---

### 边界情况

- 当输入已支持站点但格式异常时，系统应返回错误提示，不崩溃。
- 当MetaTube provider目录下存在与现有实现重名的提取器时，优先保留现有实现。
- 当新站点接口变更导致提取失败时，系统应有日志和错误提示。

## 需求 *(必填)*

### 功能需求

- **FR-001**: 系统必须移植MetaTube provider目录下所有尚未支持站点的元数据提取器。
- **FR-002**: 移植时不得覆盖/影响现有元数据提取器实现。
- **FR-003**: 用户输入新站点视频链接时，系统能自动识别并调用对应提取器，返回结构化元数据。
- **FR-004**: 移植代码需结构清晰，便于后续扩展和维护；新提取器继承现有基类，通过现有自动发现机制接入，不引入新注册方式。
- **FR-005**: 系统必须对不支持或异常站点返回明确错误提示。
- **FR-006**: 移植实现方式为参考 MetaTube Go 源码，用 Python 重写 HTTP 请求与页面解析逻辑，不依赖 Go 运行时或二进制调用。
- **FR-007**: 移植目标为 MetaTube provider 目录下所有站点，一次性全量完成（跳过与现有提取器重名的站点）。

### 关键实体 *(如果功能涉及数据则包含)*

- **元数据提取器**: 负责从特定站点页面/接口提取结构化元数据的组件，属性包括：站点标识、支持的URL模式、提取逻辑等。
- **元数据**: 结构化信息，如标题、演员、标签、时长、发布日期等。所有移植提取器必须严格对齐现有元数据模型字段（字段名与类型），站点不支持的字段填 `None`，不得引入新字段。

## 成功标准 *(必填)*

### 可衡量的结果

- **SC-001**: 新增不少于5个MetaTube provider目录下的主流站点元数据提取能力。
- **SC-002**: 所有移植提取器通过功能测试（单元测试使用 Mock 响应，纳入 CI；集成测试可选访问真实站点，不强制 CI），能正确返回结构化元数据。
- **SC-003**: 现有元数据提取器功能无回归，测试全通过。
- **SC-004**: 不支持或异常站点返回明确错误提示，用户满意度提升。

## Clarifications

### Session 2026-04-13

- Q: MetaTube SDK 是 Go 语言编写的，"移植"的含义是什么？ → A: 参考 Go 源码，用 Python 重写相同的 HTTP 请求 + 页面解析逻辑（选项 A）。
- Q: 此次移植需要覆盖哪些站点？ → A: 全量移植 provider 目录下所有站点，一次性完成（重名站点保留现有实现）。
- Q: 新移植的提取器如何接入现有系统？ → A: 沿用现有基类 + 自动发现机制，新提取器继承同一基类，不引入新注册机制。
- Q: 移植后的提取器测试策略是什么？ → A: 单元测试使用 Mock/录制响应（不依赖真实网络，纳入 CI）；集成测试可选地访问真实站点，不强制列入 CI 通过条件。
- Q: 移植的提取器返回字段是否需要与现有元数据模型严格对齐？ → A: 严格对齐现有模型，缺失字段填 `None`，不引入新字段，下游无需改动。
