# PAVOne v0.3.1 开发计划 — 元数据插件架构重构

## 📋 版本概述

本版本聚焦于**元数据插件架构的重构与代码精简**，通过提取公共基类、消除重复代码来大幅降低维护成本。

当前 34 个元数据插件总计约 6,900 行代码，存在大量结构性重复。预计重构后可减少约 1,400+ 行（~20%），同时提升可读性和可维护性。

**主要目标**:

1. 提取公共 HTML 解析基类，消除重复的模板方法
2. 将分散在各插件中的工具函数统一到基类
3. 统一 `extract_metadata` 的 try/except/logging 流程
4. 简化新插件的开发成本（从 ~180 行降到 ~100 行）

---

## 📊 问题分析

### 当前架构的代码重复统计

| 重复类别 | 每插件重复行数 | 34 插件总重复 | 说明 |
|----------|-------------|-------------|------|
| `_abs()` 静态方法 | ~7 行 | ~238 行 | 相对 URL 转绝对 URL，几乎逐字相同 |
| `_parse_runtime()` 静态方法 | ~7 行 | ~238 行 | "XX分" / "H:M:S" → 分钟数，仅少数变体 |
| `_parse_date()` 静态方法 | ~6 行 | ~204 行 | 日期字符串 → YYYY-MM-DD 格式化 |
| `extract_metadata` 模板 | ~15 行 | ~510 行 | resolve → fetch → parse → error handling |
| `_resolve` 方法模式 | ~10 行 | ~340 行 | URL 判断 + ID → URL 构造 |
| **合计可节省** | **~45 行/插件** | **~1,530 行** | |

### 当前架构层次

```
BasePlugin (plugins/base.py)
  └─ MetadataPlugin (plugins/metadata/base.py) — 仅定义 2 个抽象方法
       ├─ SodMetadata (175 行) — 重复 _abs, _parse_runtime, _parse_date, extract_metadata 模板
       ├─ DugaMetadata (192 行) — 同上
       ├─ DahliaMetadata (150 行) — 同上
       ├─ C0930Metadata (164 行) — 同上 + 独有 _parse_iso_duration
       └─ ... (34 个插件都重复以上模式)
```

### 对比 metatube-sdk-go 的做法

metatube 中：
- `_abs` / `_parse_runtime` / `_parse_date` 等是**包级公共函数**，所有 provider 共享
- 错误处理由**调用层统一封装**，provider 只需实现核心解析逻辑
- provider 直接返回结构体，无 Builder 中间层

## 阶段一：提取公共 HTML 解析基类

### 1.1 创建 `HtmlMetadataPlugin` 中间基类

在 `MetadataPlugin` 和具体插件之间插入一个新基类，封装 HTML 解析类插件的通用流程。

**目标架构**:

```
MetadataPlugin (已有, 抽象基类)
  ├─ HtmlMetadataPlugin (新增, HTML 解析通用基类)
  │    ├─ SodMetadata
  │    ├─ DugaMetadata
  │    ├─ DahliaMetadata
  │    ├─ JavbusMetadata
  │    └─ ... (大部分 HTML 解析插件)
  ├─ JsonLdMetadataPlugin (新增, JSON-LD 解析通用基类)
  │    ├─ C0930Metadata
  │    ├─ H0930Metadata
  │    ├─ H4610Metadata
  │    └─ HeyzoMetadata (混合: JSON-LD + HTML fallback)
  ├─ ApiMetadataPlugin (新增, API/JSON 解析通用基类)
  │    ├─ ModelmediaasiMetadata
  │    ├─ MuramuraMetadata
  │    ├─ PacopacomamaMetadata
  │    └─ TheporndbMetadata
  └─ (特殊插件保持直接继承 MetadataPlugin)
```

**`HtmlMetadataPlugin` 核心内容**:

```python
class HtmlMetadataPlugin(MetadataPlugin):
    """HTML 解析类元数据插件的公共基类"""

    # ── 模板方法: 统一 extract_metadata 流程 ──

    def extract_metadata(self, identifier):
        """模板方法: resolve → fetch → parse, 统一错误处理"""
        try:
            movie_id, page_url = self._resolve(identifier)
            if not movie_id or not page_url:
                self.logger.error(f"无法解析 identifier: {identifier}")
                return None
            resp = self._fetch_page(page_url)
            soup = BeautifulSoup(resp.text, "lxml")
            return self._parse(soup, movie_id, page_url)
        except requests.RequestException as e:
            self.logger.error(f"HTTP 请求失败: {e}")
            return None
        except Exception as e:
            self.logger.error(f"提取元数据失败: {e}", exc_info=True)
            return None

    def _fetch_page(self, url):
        """获取页面，子类可覆写以添加 cookies/headers"""
        return self.fetch(url, timeout=30)

    # ── 抽象方法: 子类必须实现 ──

    @abstractmethod
    def _resolve(self, identifier) -> Tuple[str, Optional[str]]:
        """将 identifier 解析为 (movie_id, page_url)"""
        ...

    @abstractmethod
    def _parse(self, soup, movie_id, page_url) -> Optional[MovieMetadata]:
        """从 BeautifulSoup 对象解析元数据"""
        ...

    # ── 公共工具方法 ──

    @staticmethod
    def _abs(url, base):
        """相对 URL 转绝对 URL"""
        ...

    @staticmethod
    def _parse_runtime(text):
        """解析时长文本 ('XX分', 'H:M:S', 'XXmin') → 分钟数"""
        ...

    @staticmethod
    def _parse_date(text):
        """解析日期文本 → 'YYYY-MM-DD' 格式"""
        ...

    @staticmethod
    def _parse_iso_duration(text):
        """解析 ISO 8601 时长 (PT1H30M) → 分钟数"""
        ...

    @staticmethod
    def _get_tag_attr(tag, attr):
        """安全获取 BS4 tag 属性并转为 str (消除 _AttributeValue 类型问题)"""
        val = tag.get(attr) if tag else None
        return str(val) if isinstance(val, str) else None
```

## ⚠️ 风险与注意事项

1. **行为兼容性**: 重构不能改变任何插件的提取结果，需要有完善的测试覆盖
2. **特殊插件**: 少数插件有独特流程（如 `avbase` 的 API+HTML 双通道，`fanza` 的 `__NEXT_DATA__` 解析），需保留灵活性
3. **渐进式迁移**: 不要一次性迁移所有插件，分批进行，每批确认测试通过
4. **向后兼容**: `MetadataPlugin` 基类接口不变，未迁移的插件仍可正常工作
